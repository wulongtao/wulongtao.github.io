本文主要内容是对于部分源码的理解还有在使用Nacos时遇到的一些问题汇总。

对于阅读源码，我的思路是先找到一些注册成功的日志，然后打断点，看整个调用链路。

客户端注册源码解读

在NacosServiceRegistry类的注册完成日志的代码打个断点，我们可以看到项目启动后的整个调用链路。

从头开始看，首先还是进到熟悉的refresh方法，在finishRefresh中做处理。

实际上创建完实例后，使用了spring的发布事件机制，会发布一个注册事件。

发布完事件后，实际上是给springcloud包（非alibaba）的AbstractAutoServiceRegistration类监听到了这个事件，然后NacosServiceRegistry又实现了ServiceRegistry，所以由Nacos接收到这个事件并注册。

这里可以看到，真正注册的逻辑在下面这个方法中。

com.alibaba.nacos.client.naming.NacosNamingService#registerInstance(java.lang.String, java.lang.String, com.alibaba.nacos.api.naming.pojo.Instance)

打断点进去会看到正真处理的类是NamingClientProxyDelegate，里面会先调用getExecuteClientProxy方法去选择执行注册的代理，这里比较有意思：

private NamingClientProxy getExecuteClientProxy(Instance instance) {
    return instance.isEphemeral() ? grpcClientProxy : httpClientProxy;
}

这里会判断如果是临时实例，用的是GRPC；持久化实例用的是httpClient。我觉得这里原因是因为临时实例的数据同步和健康检查是由客户端发送心跳的，这里使用长连接可以节省频繁发送心跳建立链接消耗性能的问题，并且客户端连接用的是nacos提供的包所以肯定支持GRPC协议。
但是持久化实例不一样，他主要是使用服务端探测的健康检查方式，因为客户端不会上报心跳，所以他并不知道客户端是否支持GRPC（例如mysql就不支持），所以他只能用httpClient的方式。

因为我调试时用的是临时实例，所以进到了NamingGrpcClientProxy中：

NamingGrpcClientProxy最终封装一个Request去注册服务实例

服务端源码阅读

服务端处理客户端链接请求

这里Nacos的设计思想和客户端有点像，也是通过发布订阅事件来完成的，在客户端与服务器建立连接后，客户端发送过来的请求会经过GrpcRequestAcceptor这个类的request方法。启动一个客户端时，会进入到下面这个断点（这里调试是要给断点加一个条件，否则会一直触发断点）。

然后我们关注下面这行代码

Response response = requestHandler.handleRequest(request, requestMeta);

这个代码最终会进到InstanceRequestHandler类的handle方法中，并且type是注册实例

注册实例的方法registerInstance进去，发现就是发布了很多个事件

最终在EphemeralClientOperationServiceImpl的下面代码块中，会把这个实例注册到ConcurrentHashMap里面去

InstancePublishInfo instanceInfo = getPublishInfo(instance);
client.addServiceInstance(singleton, instanceInfo);

整个事件发布和处理的模型图大致如下：

  1. 比如一个服务注册事件，调用NotifyCenter的静态方法开始发布一个事件，想要通过通知中心发布一个服务注册的消息
  2. 该事件对应一些发布者，这时候会去map里面找这个事件的发布者
  3. 这个事件发布者呢，他有一个阻塞队列queue，专门用来存储事件的
  4. 同时这个发布者也是一个线程，作为一个发布者的同时，也是作为一个线程被保存在NotifyCenter的map里面
  5. 发布者这个线程呢，会不断的从queue里面取事件
  6. 发布者取出事件后，会从它内部的一个map里面找到这个事件的所有的订阅者，并判断这个订阅者是需要同步还是异步通知

这里的事件处理器是DistroClientDataProcessor

配置中心加载源码

加载的入口在NacosConfigDataLoader类的pullConfig方法中，具体流程是通过grpc去链接，最后监听配置的改变

配置文件加载顺序源码

在这个类中的NacosPropertySourceLocator如下方法，从这段代码基本可以分析共享配置优先级<扩展配置<内部规则配置

public PropertySource<?> locate(Environment env) {
    nacosConfigProperties.setEnvironment(env);
    ConfigService configService = nacosConfigManager.getConfigService();

    if (null == configService) {
        log.warn("no instance of config service found, can't load config from nacos");
        return null;
    }
    long timeout = nacosConfigProperties.getTimeout();
    nacosPropertySourceBuilder = new NacosPropertySourceBuilder(configService,
            timeout);
    String name = nacosConfigProperties.getName();

    String dataIdPrefix = nacosConfigProperties.getPrefix();
    if (StringUtils.isEmpty(dataIdPrefix)) {
        dataIdPrefix = name;
    }

    if (StringUtils.isEmpty(dataIdPrefix)) {
        dataIdPrefix = env.getProperty("spring.application.name");
    }

    CompositePropertySource composite = new CompositePropertySource(
            NACOS_PROPERTY_SOURCE_NAME);
        
    loadSharedConfiguration(composite); // 加载共享配置
    loadExtConfiguration(composite); // 加载扩展配置
    loadApplicationConfiguration(composite, dataIdPrefix, nacosConfigProperties, env); // 加载内部规则配置
    return composite;
}

数据库链接不上的问题（No Database Set）

可以试试在配置文件中把下面的配置注释掉

# spring.datasource.platform=mysql

然后数据库连接配置改成这样，主要加了allowPublicKeyRetrieval和换一下时区：

jdbc:mysql://127.0.0.1:3306/nacos_config?characterEncoding=utf8&connectTimeout=1000&socketTimeout=3000&autoReconnect=true&useUnicode=true&useSSL=false&serverTimezone=Asia/Shanghai&allowPublicKeyRetrieval=true
