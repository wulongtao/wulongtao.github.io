Nacos源码阅读与实战

2023-07-18
SpringCloud
610 词
  • Nacos
  • SpringCloud

本文主要内容是对于部分源码的理解还有在使用Nacos时遇到的一些问题汇总。

对于阅读源码，我的思路是先找到一些注册成功的日志，然后打断点，看整个调用链路。

客户端注册源码解读

在NacosServiceRegistry类的注册完成日志的代码打个断点，我们可以看到项目启动后的整个调用链路。

从头开始看，首先还是进到熟悉的refresh方法，在finishRefresh中做处理。

实际上创建完实例后，使用了spring的发布事件机制，会发布一个注册事件。

发布完事件后，实际上是给springcloud包（非alibaba）的AbstractAutoServiceRegistration类监听到了这个事件，然后NacosServiceRegistry又实现了ServiceRegistry，所以由Nacos接收到这个事件并注册。

这里可以看到，真正注册的逻辑在下面这个方法中。

com.alibaba.nacos.client.naming.NacosNamingService#registerInstance(java.lang.String, java.lang.String, com.alibaba.nacos.api.naming.pojo.Instance)

打断点进去会看到正真处理的类是NamingClientProxyDelegate，里面会先调用getExecuteClientProxy方法去选择执行注册的代理，这里比较有意思：

private NamingClientProxy getExecuteClientProxy(Instance instance) {
    return instance.isEphemeral() ? grpcClientProxy : httpClientProxy;
}

这里会判断如果是临时实例，用的是GRPC；持久化实例用的是httpClient。我觉得这里原因是因为临时实例的数据同步和健康检查是由客户端发送心跳的，这里使用长连接可以节省频繁发送心跳建立链接消耗性能的问题，并且客户端连接用的是nacos提供的包所以肯定支持GRPC协议。
但是持久化实例不一样，他主要是使用服务端探测的健康检查方式，因为客户端不会上报心跳，所以他并不知道客户端是否支持GRPC（例如mysql就不支持），所以他只能用httpClient的方式。

Nacos部署遇到的一些问题

数据库链接不上的问题（No Database Set）

可以试试在配置文件中把下面的配置注释掉

# spring.datasource.platform=mysql

然后数据库连接配置改成这样，主要加了allowPublicKeyRetrieval和换一下时区：

jdbc:mysql://127.0.0.1:3306/nacos_config?characterEncoding=utf8&connectTimeout=1000&socketTimeout=3000&autoReconnect=true&useUnicode=true&useSSL=false&serverTimezone=Asia/Shanghai&allowPublicKeyRetrieval=true
